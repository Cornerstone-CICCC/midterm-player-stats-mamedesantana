# Reflection

I built a full-stack Player Performance Explorer using Astro for the front end, Node.js and Express for the API, and PostgreSQL through the `pg` Pool. The application allows users to browse player-match performances with server-side pagination, change the number of rows per page, search, filter, sort, open a full detail view, create new records, edit existing records, and delete records. I also built a rankings page that aggregates each player's matches, goals, assists, and average rating.

The original CSV is a flat dataset, meaning that player information, match information, and match-specific statistics are repeated in every row. The setup script normalizes the data into three tables. The `players` table stores attributes that belong to a player, including the player's name, team, position, nationality, jersey number, club, and tournament totals. The `matches` table stores match-level attributes such as date, stadium, city, and tournament stage. The `performances` table stores one player's statistics in one match and includes the foreign keys `player_id` and `match_id`. This design reduces duplicated player and match data and keeps related information consistent.

The most challenging parts were building dynamic but safe filtering and sorting queries, calculating pagination, and connecting Astro forms to Express endpoints. Query values are parameterized with placeholders such as `$1`, while sortable column names are selected from an allow-list so that user input is never concatenated directly into SQL. With more time, I would add automated tests, stronger form validation, authentication, charts, and more responsive mobile layouts.
